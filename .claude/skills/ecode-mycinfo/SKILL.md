# E10 ecode 自有案例库维护

## 概述

将用户提供的 ecode 前端 JS 代码或功能文件夹，从**业务 + 技术双视角**解析并整理为 `knowledge-base/frontend/mycase/` 下的标准化案例。

## 案例库位置

```
knowledge-base/frontend/mycase/
├── INDEX.md                    # 案例索引（业务场景 + 技术实现 双列表格）
├── 案例名1/
│   ├── entry.js                # 完整代码
│   └── README.md               # 业务场景 + 技术实现 双章节
└── 案例名2/
    ├── ...
```

## 解析规则

### 双视角分析

拿到代码后必须从两个角度分析：

**1. 业务视角**：回答"用户要做什么"
- 页面类型和位置（EB 列表右上角 / EB 表单按钮 / 流程表单提交前 / 独立页面 等）
- 区分概念：EB 列表（数据列表页） ≠ EB 表单 / 流程表单（需要填写的 form 形式）
- 操作流程（点击 X → 弹窗 Y → 填写 Z → 提交 → 等待 → 结果）
- 业务目的（触发什么工作流、完成什么任务、解决什么问题）

**2. 技术视角**：回答"代码怎么实现的"
- 完整代码链路，精确到函数名、接口路径、组件名
- 核心技术手段（ReactDOM.render 命令式弹窗 / regOvComponent 复写 / entry.js 注册 等）
- 关键 API 调用（ESB 接口路径、WeFormSDK 方法等）
- 特殊处理（异常安全、动画实现、去重策略等）

### 代码分析要点

阅读代码时重点提取：

| 关注点 | 提取内容 |
|--------|---------|
| **入口** | 按钮事件绑定方式（window.xxx / eb 按钮配置 / regOvComponent） |
| **UI 组件** | 使用的 @weapp/ui 组件（Dialog/DatePicker/Button 等） |
| **弹窗方式** | ReactDOM.render 命令式 / 组件声明式 / window.weappUi.Dialog.message |
| **接口调用** | request/axios、接口路径、请求参数 |
| **数据流** | 闭包变量 / React state / 表单 SDK 取值 |
| **特殊逻辑** | loading 状态、异常处理、校验逻辑 |

## 执行流程

### 第 1 步：理解代码

阅读用户提供的 JS 代码，理解：
- 这是什么业务场景
- 代码入口在哪，怎么触发的
- 核心交互流程是什么
- 用了哪些技术手段

如果用户提供的是文件夹，先 `ls` 看结构，再读入口文件和核心组件。

### 第 2 步：命名案例文件夹

格式：`业务场景简述-核心技术点/`

- 用中文描述，简洁有辨识度
- 例如：`薪资计算ESB工作流-Loading弹窗/`
- 放在 `knowledge-base/frontend/mycase/` 下

### 第 3 步：写入 entry.js

将完整代码写入 `entry.js`，顶部加注释标注：

```js
// ==========================================
// 业务场景简述
// 来源：mycase（自有案例，生产验证）
// ==========================================
```

**不修改代码逻辑**，保持原样。

### 第 4 步：生成 README.md

按模板生成，分两大章节：

```markdown
# 案例标题 — 一句话说明

## 业务场景

[按用户操作步骤叙述，具体到页面位置、操作流程、业务目的]

## 技术实现

### 整体架构

[ASCII 流程图，展示完整代码链路]

### 技术要点

| 要点 | 说明 |
|------|------|
| ... | ... |
```

### 第 5 步：更新 INDEX.md

在 `knowledge-base/frontend/mycase/INDEX.md` 的表格中新增一行：

```markdown
| N | [案例名](./案例名/) | **业务**：详细描述用户操作流程... | **技术**：代码链路 + 接口 + 组件... |
```

业务场景列和技术实现列都要详细，避免仅写简单标签。

### 第 6 步：更新 ecode-developer skill（可选）

如果新案例涉及新技术模式（之前 skill 未覆盖的），更新 `.claude/skills/ecode-developer/SKILL.md` 的核心文件速查表。

一般情况下不需要这步，因为 mycase/INDEX.md 已被 skill 引用。

## 输出格式

完成后向用户报告：

```
## 案例入库完成

- 案例名称：[名称]
- 文件夹：`knowledge-base/frontend/mycase/xxx/`
- 文件：entry.js + README.md
- INDEX.md：第 N 行

### 业务场景
[一句话描述]

### 技术实现
[一句话描述]
```

## 注意事项

1. **代码原样保留**：不修改用户提供的代码逻辑，仅加顶部注释
2. **双视角不可缺一**：业务和技术两个角度都要覆盖，不能只写技术
3. **描述要具体**：避免"按钮触发ESB"这种笼统描述，要写"EB 表单列表右上角自定义按钮 → 点击后弹窗..."
4. **区分列表和表单**：EB 列表是数据列表页（右上角加按钮），EB 表单是需要填写的 form
5. **索引及时更新**：每次添加案例必须同步更新 INDEX.md
6. **文件夹命名规范**：中文 + 简短，体现业务+技术特征
