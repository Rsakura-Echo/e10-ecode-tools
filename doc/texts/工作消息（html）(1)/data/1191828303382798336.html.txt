一、 功能背景
员工间沟通时经常存在以下情况，如普通员工向高管频繁发送无关消息可能干扰决策效率，跨部门敏感信息泄露风险增加，或特定岗位（如财务、法务）需避免非必要沟通。传统通讯工具缺乏细粒度权限控制，导致企业面临信息泄露风险与沟通效率低下的双重挑战。
二、 功能价值
1. 精准权限控制：支持多维度配置，降低信息干扰。
2. 系统与个人规则结合：管理员全局管控与员工自定义规则互补，满足不同管理需求。
3. 合规与风险规避：限制敏感操作，防止信息泄露，满足审计与监管要求。
4. 避免无效沟通：通过场景化权限（如临时单聊），平衡紧急协作与长期权限管控，减少无效沟通。
三、 功能概述
1. 聊天权限限制范围包括：单聊 / 拉保护对象进群 / 群内@保护对象 / 必达保护对象 / 与保护对象音视频通话；
2. 聊天权限规则分为：系统级预设规则和前台个人级自定义规则，个人级别高于系统级别；
3. 个人级别：保护对象可在前台限制对象的个人详情页查看聊天限制情况；可在偏好设置中查看系统级为其预设的规则及配置个人级自定义规则（不允许单聊/不允许拉进群/不允许群内@/不允许必达/不允许音视频通话）；
4. 系统级预设规则：管理员可在后台预设规则，规则中包含保护对象、限制对象及限制哪些维度（不允许单聊/不允许拉进群/不允许群内@/不允许必达/不允许音视频通话）。

五、 功能详细说明
1、后台设置-聊天权限设置-基础设置
① 启用聊天权限控制：默认关闭，开启后，pc端前台偏好设置—显示聊天限制配置入口，app端设置—消息设置—显示聊天限制配置入口，按照配置的权限规则生效，关闭表示不限制聊天，前台权限入口不显示。
② 单聊权限的审批人：默认为保护对象，也可设置指定人员进行审批；负责审批是否通过限制对象申请的单聊权限，批准后可按照配置的规则范围聊天；
当规则设置了允许申请单聊时，限制对象可通过点击会话中灰色消息提示“发送申请”，根据后台设置的审批人，审批人会收到系统消息提醒并可选择同意或拒绝。
比如有的公司是需要保护对象进行审批即可，而有的是需要经过人事审批才可进行沟通。
③ 允许添加自定义权限的用户：聊天权限控制包括系统级权限及个人级权限；系统级权限是后台管理员预设的规则，普通用户无法修改；
个人级权限是用户在前台自定义的规则，可限制哪些人与自己聊天。管理员可后台配置哪些人有权限自定义个人规则，无权限则只能查看，不能新建个人规则。
比如有的公司是允许全体员工都可以配置限制谁与自己聊天，而有的公司则只允许一些高管才有权限去限制谁与自己聊天。
2、后台设置-聊天权限设置-系统级配置
① 系统预设规则：规则中包含保护对象、限制对象及限制哪些维度（不允许单聊/不允许拉进群/不允许群内@/不允许必达/不允许音视频通话）。
比如可以提前为一些高管预设好规则，就不用麻烦高管自己去一个个配置了。
保护对象：保护对象指不被别人打扰，如高层领导。可设置包含/排除人员、部门、分部、角色、岗位、所有人等，也可在详情中根据人员安全级别进行配置。
限制对象：限制对象指限制不能打扰别人，如普通员工。可设置包含/排除人员、部门、分部、角色、岗位、所有人等，也可在详情中根据人员安全级别进行配置。
不允许单聊：勾选后显示允许申请单聊、允许单聊回复开关；
允许申请单聊开关，默认关闭，开启后可设置X天内只能申请一次，最少1天，申请单聊的类型包括长期单聊和临时单聊，临时单聊可设置申请通过X小时后恢复聊天限制。
不允许拉进群：勾选后显示仅支持同意后进群（默认未选中）；不允许被拉进群，被限制方发起群聊包含保护方时，提示XX人员邀请失败；
不允许被拉进群同时选中仅支持同意后入群，则被限制方发起群聊包含保护方时，提示邀请XX人员已发送申请，对应人员收到系统消息，可点击同意/拒绝（群聊消息转话题、动态规则入群的不做限制）。
不允许群内@：勾选后不允许限制对象@保护对象，包括：
@保护方个人、包含"保护方"的群身份；
限制方@“保护方”，发送至会话的消息为普通消息体，保护方也不会收到@高亮提醒，且有灰色消息提示：你无权限@XXX（@全体成员不做限制）。
不允许必达：勾选后不允许“限制对象”给“保护对象”发送必达；包括消息转必达、聊天工具栏发送必达、群消息未读详情转必达、会话顶部发送必达等场景。
不允许音视频通话：勾选后不允许"限制对象"给"保护方"发送音视频通话，包括单聊/群聊聊天工具栏发起、会话顶部发起音视频通话等场景。
② 权限查询：搜索中选择"保护对象"及"限制对象"，即可展示二者间生效规则，包括最终生效规则情况、用户个人级规则情况及系统预设规则情况。
③ 系统级查询：搜索中选择"保护对象"/"限制对象"，可查询系统预设规则中包含"保护对象"/"限制对象"的相关数据。
④ 重置申请状态：会将该组织下所有已经申请单聊并通过的权限进行重置清除，恢复为未申请的情况。
比如普通员工申请与高管沟通，高管同意申请后，后面又想限制时，就可以重置申请状态，此时需要再次申请并通过才可继续沟通。
六、更多场景示例
1、高管保护
普通员工向董事长/总经理等高管汇报时，没有单聊权限，需提交单聊申请，由指定人审批并通过后才可进行沟通汇报。
2、跨部门协作
销售部员工可申请与技术部员工临时沟通项目进度，但禁止长期私聊、必达及进行音视频通话。
3、群聊管控
禁止市场团队拉技术同事进群并在群内@技术同事，避免干扰技术同事日常工作，影响效率。
4、敏感岗位隔离
法务人员仅允许与同部门同岗位进行日常沟通交流，与其他人员皆不可进行沟通。
七、上下游关联功能点
1. 我的好友：被组织机构应用分权隔离的本组织人员，若好友权限为“仅聊天”时，此时若再限制不允许单聊的，则依然无法正常聊天，二者功能皆生效。更多详情点击查看
2. 消息及聊天权限：当压根没有消息权限/聊天权限/创建必达权限时，不管是否有聊天限制，都是无法沟通的。更多详情点击查看
八、注意点
1. 当系统级规则与个人级规则冲突时，按照“最严格原则”生效​（例如系统级禁止私聊 + 个人级允许临时聊天 → 最终禁止私聊）；
2. 开启允许申请单聊时，限制申请次数不要太低，防止恶意刷申请，对保护方造成困扰；
3. 同一保护对象及限制对象尽量不要创建多个规则，会导致性能问题且不太易于理解；
4. 目前限制规则仅对内部成员生效，外部用户暂不生效。
